## 一、项目背景与目标

### 1.1 现状与痛点

当前部门围绕 O32 投资交易系统的日常运维工作已沉淀了一批标准化处理能力（以 AI 助手 Skill 及 exe 小工具形式承载）。存在以下共性痛点：

1. **工具分散、依赖个人**：核对工具以单机 exe、AI Skill 等形式分散存在，规则与结果都落在个人电脑上，无法多人共享与沉淀。
2. **规则散落**：信托计划代码映射规则（21 条）、大宗产品清单（11 个）、I9 过滤规则、差异阈值（1%）、模糊匹配阈值（0.5）等以 JSON 配置/代码常量形式分散维护，存在口径漂移风险。
3. **结果无归档**：核对结果 Excel 散落本地目录，靠文件名区分版本，无法按日期、产品、差异状态检索回溯与跨日对比。
4. **数据获取靠人工导出**：每日需人工从 O32 / 估值系统导出文件再执行核对，重复性劳动占比高，具备数据库只读直连后自动取数的条件。
5. **权限与审计缺失**：单机工具无用户体系与操作审计，不满足内控要求。

### 1.2 建设目标

建设一套 **B/S 架构的日常运维平台**，将现有能力项逐个落地为平台功能菜单：

- **功能菜单化**：每个能力项对应平台一个独立功能菜单，浏览器完成"取数（上传/直连）→ 执行核对 → 查看/下载结果"全流程。
- **规则配置化**：映射规则、大宗产品清单、科目取价规则、阈值等全部后台配置，变更有记录、可追溯。
- **取数双模式**：支持文件上传与数据库只读直连两种方式获取核对数据；直连模式支持 Oracle、MariaDB 等主流数据库。
- **任务自动化**：每日/定期核对支持定时调度，直连模式下可实现零人工干预。
- **结果资产化**：核对报告统一存储、可检索、可跨日对比。
- **账号体系化**：默认本地账号密码登录，提供独立用户维护菜单；保留 AD 域对接能力，可通过配置切换。
- **口径一致性**：M1 核对逻辑与输出格式与已交付 exe 小程序完全一致，确保平滑过渡与交叉验证。

### 1.3 建设原则

- **先核对、后扩展**：优先落地高频、规则明确的核对类功能。
- **规则与代码分离**：业务口径一律配置化，配置变更不需要改代码、不需要重启服务（热生效）。
- **数据不出内网**：平台部署于内网，文件与结果本地存储。
- **只读红线**：数据库直连能力严格只读，平台不具备也不提供任何写库能力。
- **输出一致性**：M1 输出与 exe 小程序输出逐行一致（列结构、颜色标注、统计口径、文件命名）。
- **文档与代码独立**：文档与代码分文件夹存放（路径独立、整洁管理）；文档按版本存档，接口契约文档化（OpenAPI）。
- **交付免环境**：目标服务器与操作终端无开发环境、不允许额外安装插件或运行时，交付物必须全量预编译、解压即用。

### 2.3 M1 小程序现状摘要（平台 M1 的对齐基准）

已交付 exe 小程序（`fund_reconciler.py` v1.0 命令行版 + `fund_reconciler_gui.py` v2.1 图形版），其逻辑即平台 M1 的验收基准：

- **处理流水线（11 步，顺序固定）**：读取两表 → 提取关键列 → 重命名映射 → 映射去重 → I9 过滤 → 精确匹配 → 模糊匹配 → 计算差异 → 排序（未匹配置底，其余按最大差异降序）→ 标记大宗产品 → 生成 Excel
- **大宗产品机制**：内置 11 个大宗产品代码（AZ0206/AZ0205/AZ0221/AZ0230/AZ0130/AZ0160/AZ0208/AZ0236/AZ0210/AZ0151/AZ0207），标橙色、差异原因注明"大宗产品无需核对"、**不参与差异超阈值统计**
- **配置与代码分离**：`fund_reconciler_config.json` 承载 rename_map / bulk_products / diff_threshold / similarity_threshold，配置热更新、改配置不重新编译
- **输入校验**：基金资产表列数 <28 或净值查询表列数 <9 时明确报错；检测文件疑似选反并告警
- **GUI 操作流**：选择基金资产表 → 选择净值查询表 → 选择输出路径（默认桌面，目录自动生成文件名）→ 开始核对（进度条+实时日志）→ 完成弹窗（统计摘要）→ 一键打开结果文件

### 3.2 M1 基金资产与净值核对

**对齐基准：逻辑要求与操作要求均与 exe 小程序（fund_reconciler v1.0 / GUI v2.1）一致。**

#### 3.2.1 核对逻辑（与小程序逐点一致）

| 编号 | 需求 | 说明 |
|---|---|---|
| M1-F1 | 数据读取 | 读取基金资产表与净值查询表（.xls/.xlsx）；基金资产表关键列位：组合编号 Col2、组合名称 Col3、基金资产净值 Col11、基金总资产 Col27、信托计划代码 Col19；净值查询表关键列位：产品名称 Col0、查询资产净值 Col2、查询总资产 Col4、信托计划代码 Col8 |
| M1-F2 | 输入校验 | 基金资产表列数 <28、净值查询表列数 <9 时阻断执行并给出明确中文错误提示（含"是否文件选反"引导）；检测到两表列数特征疑似选反时告警 |
| M1-F3 | 预处理流水线 | 严格按序执行：**①重命名映射 → ②映射去重（同代码下删原始记录、留映射记录）→ ③I9 过滤（同代码多条且存在非 I9 时仅保留非 I9 记录）** |
| M1-F4 | 精确匹配 | 以信托计划代码为 Key，基金资产表 LEFT JOIN 净值查询表 |
| M1-F5 | 模糊匹配 | 未匹配记录按 组合名称 vs 产品名称 做 SequenceMatcher 相似度匹配，阈值默认 0.5；每条净值记录仅被匹配一次；记录相似度（保留 4 位小数） |
| M1-F6 | 差异计算 | 总资产差值/差值百分比、资产净值差值/差值百分比，分母取净值查询表数据，除零保护 |
| M1-F7 | 排序 | 未匹配记录置底，其余按最大差异百分比降序 |
| M1-F8 | 大宗产品标注 | 信托计划代码命中大宗产品清单的记录：标橙色、差异原因写"大宗产品无需核对"、**不参与差异超阈值统计**（差异统计口径 = 非大宗且已匹配记录） |
| M1-F9 | 统计摘要 | 总记录数、精确匹配、模糊匹配、未匹配、大宗产品数、差异>阈值%数量（非大宗）——统计口径与小程序完全一致 |

#### 3.2.2 Excel 输出（与小程序输出逐行一致）

| 项 | 规格 |
|---|---|
| Sheet | 单 Sheet「核对结果」 |
| 列顺序（15 列固定） | 组合编号 → 组合名称 → 信托计划代码 → 产品名称 → 匹配方式 → 匹配相似度 → 基金总资产 → 查询总资产 → 总资产差值 → 总资产差值百分比(%) → 基金资产净值 → 查询资产净值 → 资产净值差值 → 资产净值差值百分比(%) → 差异原因 |
| 颜色标注 | 🟠 大宗产品：橙色 #FFC000；🔴 差异>阈值：浅红 #FFCCCC；🟡 未匹配：浅黄 #FFFF99；表头：#B4C7DC + 粗体居中 |
| 数字格式 | 金额列 `#,##0.00`；百分比列 `0.00` |
| 其他 | 全表细边框、冻结首行（A2）、列宽按小程序规格 |
| 文件命名 | `基金资产与净值核对结果_YYYYMMDD.xlsx`；归档内同一业务日期多次执行追加时间戳/版本号，不覆盖 |

#### 3.2.3 操作流程（B/S 化，操作体验对齐小程序）

| 步骤 | 小程序操作 | 平台操作 |
|---|---|---|
| 1 | 浏览选择基金资产表 | 上传/选择基金资产表（文件模式）或选择数据源与查询模板（直连模式） |
| 2 | 浏览选择净值查询表 | 上传/选择净值查询表（文件模式）或选择数据源与查询模板（直连模式） |
| 3 | 选择输出路径 | ——（B/S 无需选路径，结果自动归档，页面上提供下载） |
| 4 | 开始核对，进度条+实时日志 | 执行任务，页面展示进度条与实时日志（服务端日志流式推送） |
| 5 | 完成弹窗显示统计摘要 | 结果页展示统计摘要卡片 + 差异明细表格（行内颜色与 Excel 一致） |
| 6 | 一键打开结果文件 | 一键下载/在线预览结果 Excel |

- 执行前页面展示当前生效配置摘要（映射规则条数、大宗产品数、差异阈值、相似度阈值）；支持本次执行临时调整阈值（不落库）。
- 支持多日期批量核对：一次上传多日期文件，自动按日期分组、逐日独立核对、分别生成报告。

### 3.3 M2 基金估值价格核对

| 编号 | 需求 | 说明 |
|---|---|---|
| M2-F1 | 数据获取 | 系统端：新综合信息查询_基金证券（第 1 行起，过滤末尾无代码汇总行）；财务端：证券投资基金估值表（第 4 行为列头）；支持文件上传与 DB 直连两种模式；支持多产品（如 6301、6302）批量处理，**按产品分别生成独立报告** |
| M2-F2 | **科目取价规则（配置化）** | **不固定 1101 取市价、1501 取单位成本**。科目取价规则在【系统管理-系统配置】菜单维护：规则项 = 科目前缀 + 取价字段（如 市价/单位成本，随估值表列扩展）+ 科目说明，支持新增、修改、停用。默认预置 1101→市价、1501→单位成本两条 |
| M2-F3 | 估值表明细提取 | 按生效的科目取价规则逐条提取：科目前缀过滤 → 仅保留数量列有值的叶子节点 → 证券代码取科目代码后 6 位 → 按规则取对应价格字段 |
| M2-F4 | 两步匹配 | P1 证券代码精确匹配 → P2 证券名称模糊匹配 |
| M2-F5 | 差异判定 | 价差 <0.0001 一致（绿）；≥0.0001 差异（红）；单边数据标橙 |
| M2-F6 | 科目类型标注 | 报告中标注每条记录的来源科目前缀及取价字段（取自配置，如"1101-交易性金融资产（取市价）"） |
| M2-F7 | 结果解读提示 | 报告自动生成口径说明；摊余成本类科目（如 1501）差异系口径差异属正常的提示语随规则可配置 |
| M2-F8 | Excel 输出 | 列：证券代码/证券名称/系统持仓/系统估值价格/估值表数量/估值表价格/价格差异/差异状态/科目类型/匹配方式；底部汇总（总记录、一致、差异、单边） |

## 五、边界与约束

1. 平台只做**核对、呈现、归档**，不回写 O32 生产库及任何业务库；数据库直连能力严格只读。
2. 银行间 ID 匹配结果（更新后基金属性表）为导入 O32 的参考文件，导入动作由操作员在 O32 客户端人工完成。
3. 涉及投资数据，平台不提供任何投资建议类功能。
4. exe 小程序保留作为应急兜底工具（平台不可用时可单机执行，两者输出一致保证结果可比）。
5. 目标服务器与操作终端**没有开发环境，且不允许在终端上额外安装任何插件或运行时**；平台所需全部运行时、依赖库、数据库驱动必须在交付包中预编译/预置。
6. **测试样本必须脱敏**：黄金样本及一切测试数据均使用脱敏构造数据（基于真实文件的结构与数据特征构造，产品名称、金额等敏感信息全部替换），不含任何真实生产数据。
7. 平台使用规模限定 **5 人以内**（据此采用内嵌 SQLite 的低并发设计）。
8. M1 核对范围维持当前 Skill/小程序能力逻辑，不新增排除类产品清单等额外功能。
9. M2 直连估值系统的表结构文档后期由业务方提供（二期启动前落实）；在此之前 M2 走文件模式。

## 六、验收标准（摘要）

| 编号 | 验收项 | 标准 |
|---|---|---|
| V1 | M1 输出一致性 | 同一对输入文件，平台输出与 exe 小程序输出逐行一致（匹配结果、差异值、颜色标注、统计口径）；大宗产品标注与剔除统计行为一致 |

| V6 | 规则配置生效 | 修改映射规则/大宗清单/阈值后，新执行任务按新规则输出，历史归档不受影响；配置变更有审计记录 |

| V9 | 归档策略 | 默认全量保留；配置保留 N 天后自动清理仅删文件、保留索引记录 |
| V10 | 免环境交付 | 交付包在**无 Python/Node/任何运行时的干净 Windows 环境**解压即用：服务启动、页面访问、M1 核对全流程通过；全程无需安装任何插件 |
| V11 | 路径隔离 | 源码与交付布局中文档、前端、后端分属独立文件夹（docs/、client/、server/）；前后端各自独立构建成功，接口契约文档随版本存档 |
