# 版本管理规范

> 适用范围：安联资管运维管理平台（O32-OpsPlatform）全部代码、文档、部署包。
> 生效日期：2026-07-24（v0.7.2 起执行）。

## 一、版本号规则（SemVer 简化版）

格式：`v主版本.次版本.修订号`（如 v0.7.2）

| 变更类型 | 升级位 | 示例 |
|---|---|---|
| 新增功能菜单 / 重要新能力 | 次版本（minor） | v0.6.9 → v0.7.0（新增 DBF 查看） |
| 缺陷修复 / 小优化 / 兼容性改进 | 修订号（patch） | v0.7.0 → v0.7.1（DBF 内容嗅探兼容 .713） |
| 架构级重构 / 不兼容变更 | 主版本（major） | 未来 1.0.0（正式投产定版时启用） |

## 二、版本号三处同步（缺一不可）

每次发版必须同步修改以下三处，保证"代码 = 包 = 仓库"一致：

| 位置 | 内容 | 说明 |
|---|---|---|
| `package/build.bat` | `set VERSION=X.Y.Z` | 部署包文件名 `o32-ops-platform-vX.Y.Z.zip` 的来源 |
| `package/gen_pkg_info.py` | CHANGELOG 顶部新增版本条目 | 写入部署包 `VERSION.txt`，平台"版本升级"页可见 |
| git annotated tag | `git tag -a vX.Y.Z -m "..."` | 仓库版本锚点，可随时回溯对应代码 |

## 三、发版流程（九步，顺序不可颠倒）

1. **开发**：在 `server/`、`client/` 内完成功能开发，前后端隔离，接口走 OpenAPI 契约
2. **测试**：跑通黄金样本回归与既有测试集（当前基线：18 组 849 项全绿），新增功能补测试
3. **构建**：执行 `package/build.bat`（前端 npm build → PyInstaller onedir 冻结依赖 → 组装部署包 zip）
4. **部署验证**：用"版本升级"页在线上传部署包，完成自动备份、替换、自重启
5. **端到端验证（E2E）**：浏览器实测新功能 + 回归核心链路（登录/核对任务/规则配置/审计留痕）
6. **提交**：`git commit`，message 格式 `type(scope): vX.Y.Z 一句话说明`（type ∈ feat/fix/chore/docs）
7. **打标签**：`git tag -a vX.Y.Z -m "版本说明"`
8. **推送**：`git push origin master --tags`（提交与标签一起推）
9. **汇报**：向用户交付变更说明 + 验证清单（测了什么、结果如何、未验证项如实标注）

## 四、部署包命名与留存

- 命名：`o32-ops-platform-vX.Y.Z.zip`，统一产出在 `package/dist/`
- 旧版部署包默认清理，只保留当前版；历史版本一律通过 git tag 回溯重建，不留散包
- 部署实例固定目录 `D:\O32-Ops`，`data\` 与 `app\` 物理分离；在线升级前自动备份至 `backups\`

## 五、数据连续性铁律（发版红线）

1. 用户业务数据（用户、数据源、规则、模板、任务历史、审计日志）只存 `data\` 目录，升级**永不触碰**
2. 升级方式唯一入口：平台"版本升级"页在线上传 zip，**禁止**任何形式的手工数据迁移脚本（upgrade.bat 已废弃）
3. 每次升级自动备份 `app\` 与 `data\`，异常可回滚
4. 数据库结构变更必须写迁移逻辑，随程序启动自动执行，不允许"升级后手工跑 SQL"

## 六、仓库纪律

- 远程仓库：`https://github.com/dupeng1006/ITops.git`，工作分支 `master`
- 密钥不入仓：`O32OPS_SECRET_KEY`、`.env`、`data/secret.key` 一律走环境变量或首启自动生成
- 黄金样本（`samples/golden/`）必须全脱敏虚构数据；验收基准（`samples/reference/`）只读禁改
- 文档与代码分目录管理：`docs/`（01-需求 / 02-方案 / 03-手册 / 04-测试）与 `client/`、`server/`、`package/`、`samples/` 各自独立
