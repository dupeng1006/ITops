### 1.2 技术选型

| 层 | 选型 | 选型理由 |
|---|---|---|
| 前端 | Vue 3 + Element Plus + ECharts + Vite | 中后台表单/表格生态成熟 |
| 后端 | **Python 3.11 + FastAPI** | M1 引擎可直接复用小程序已验证代码（类化、配置注入式），零口径风险 |
| 数据处理 | pandas + openpyxl + xlrd | 与小程序技术栈完全一致 |
| **多数据库访问** | **SQLAlchemy 2.x + 方言驱动**：`oracledb`（Oracle，thin 模式免装客户端）、`pymysql`（MariaDB/MySQL，纯 Python）、`pymssql`（SQL Server，wheel 自带 FreeTDS）、`psycopg2-binary`（PostgreSQL，自带 libpq） | 统一连接管理与方言抽象；**全部驱动均可随 exe 打包、目标机免安装** |
| SQL 安全校验 | `sqlparse` 语句解析白名单 | 平台侧强制只读（见 2.4） |
| 认证 | 本地账号（bcrypt 哈希）默认；`ldap3` 实现 AD 认证，配置化切换 | 不依赖域控开通周期即可上线，保留 AD 对接能力 |
| 异步任务 | APScheduler + 进程池；WebSocket 推送进度/日志 | 对齐小程序"进度条+实时日志"体验 |
| 平台数据库 | **默认内嵌 SQLite（WAL 模式，随交付包零安装）**；保留 MariaDB 切换能力（配置项） | 目标服务器无开发/安装环境，SQLite 免部署；内部平台低并发下 WAL 足够（论证见议题 7） |
| **后端交付** | **PyInstaller 打包为独立 exe**：内置 Python 3.11 运行时与全部依赖（pandas/openpyxl/xlrd/oracledb/pymysql/pymssql/psycopg2-binary/sqlparse/ldap3/bcrypt/APScheduler 等） | 目标机无需安装 Python 及任何插件/驱动 |
| **前端交付** | 构建机 `vite build` 预编译为纯静态文件（HTML/JS/CSS），由后端 exe 直接托管（StaticFiles） | 目标机无需 Node 环境；浏览器零插件 |
| 部署 | 交付 zip 一键部署包：解压 → 初始化 → NSSM（单 exe 随包）注册 Windows 服务；无需 Nginx（可选保留） | 与"无开发环境"约束匹配 |

### 2.2 核心设计一：M1 引擎直接复用小程序代码

小程序 `fund_reconciler.py` 的 `FundAssetReconciler` 已实现**构造注入式配置**（rename_map / bulk_products / diff_threshold / similarity_threshold），与平台"规则配置化"天然契合：

1. **原样封装**：将该类纳入 `engines/m1_fund_netvalue.py`（保留全部 11 步流水线与校验逻辑），平台侧只做两件事：任务执行前从规则服务加载配置注入构造器；执行后回收 `result_df`、`stats` 与生成的 Excel。
2. **GUI 版补充逻辑回移**：GUI v2.1 中的输入校验（列数不足报错、文件选反检测、输出路径容错）一并纳入引擎，保证平台校验行为与小程序一致（验收 V2）。
3. **黄金样本回归（样本强制脱敏）**：样本一律使用**脱敏构造数据**——基于真实文件的结构与数据特征构造，产品名称、金额等敏感信息全部替换，不含任何真实生产数据；以小程序对脱敏样本的输出为预期结果，固化自动化比对测试（DataFrame 逐行 + Excel 单元格颜色/格式比对）；引擎或规则代码变更后必跑，**不一致即阻断发布**。
4. **配置迁移**：`fund_reconciler_config.json` 内容（21 条映射 + 11 个大宗产品 + 两个阈值）作为初始数据导入规则配置中心；规则服务保持与该 JSON 结构的导入/导出兼容，exe 与平台可共用同一份配置口径。

### 2.3 核心设计二：数据源适配层（文件/直连双模式）

**统一抽象**：核对引擎只面向 DataFrame，不感知数据来源。

```
M1 任务 ──┬── FileAdapter（上传 .xls/.xlsx → DataFrame）
          └── DbAdapter（数据源 + 查询模板 + 业务日期参数 → DataFrame）
                      ↓
              统一进入 FundAssetReconciler（后续逻辑完全一致）
```

**DbAdapter 工作流程**：按数据源配置建立 SQLAlchemy 引擎 → 加载查询模板 SQL → 参数绑定（如 `:biz_date`）→ SQL Guard 校验 → 执行（超时/行数限制）→ 按模板字段映射转换为标准 DataFrame → 交给引擎。

### 2.4 核心设计三：只读直连的安全防线（四道）

| 防线 | 措施 |
|---|---|
| 1. 数据库侧 | 部署文档要求为平台创建**只读账号**（各库授权示例：Oracle `GRANT CONNECT, SELECT ON ...`；MariaDB/MySQL `GRANT SELECT ON db.*`；SQL Server `db_datareader`），从权限根源禁止写操作 |
| 2. 平台 SQL 白名单 | sqlparse 解析查询模板：仅允许单条 SELECT（含 WITH 开头），拦截 INSERT/UPDATE/DELETE/DDL/CALL/EXECUTE/多语句；模板保存时校验一次，执行前再校验一次 |
| 3. 执行保护 | 语句超时（默认 60s 可配）、最大返回行数（默认 100 万）、只读事务模式（Oracle `READ ONLY` 事务、MySQL 系 `SET TRANSACTION READ ONLY`） |
| 4. 凭据与审计 | 数据源密码、AD 绑定密码使用对称加密（Fernet，密钥由环境变量注入，不入库）后存储；连接测试、模板变更、每次查询执行全部审计留痕 |

**驱动说明**：Oracle 使用 `python-oracledb` thin 模式，无需在服务器安装 Oracle Client；如后续遇到 thin 模式不支持的旧版本/特性，再切换 thick 模式（配置项预留）。所有驱动均选择可随 PyInstaller 打包、目标机免安装的版本。

### 2.5 核心设计四：认证与用户体系

- **默认本地认证**：用户名+密码，bcrypt 哈希存储；JWT 会话（8 小时过期）；首次登录强制改密。
- **用户维护菜单**（管理员）：用户 CRUD、启用/停用、重置密码、角色分配（管理员/操作员/只读）。
- **AD 认证（可配置）**：认证配置页面提供开关（本地/AD）与参数（服务器、端口、BaseDN、绑定 DN/密码、用户过滤器、域账号→本地角色映射规则）；使用 ldap3 完成绑定认证，域账号首次登录自动建档并按映射赋角色；切换方式重启生效。**默认交付即为本地认证，AD 参数留空待启用**。

### 2.6 核心设计五：系统配置（科目取价规则等）

- `sys_subject_price_rule` 表：科目前缀、取价字段（如 市价/单位成本）、科目说明、口径提示语（如 1501 摊余成本差异属正常的说明文案）、启用状态、排序号。M2 引擎执行时动态加载启用规则逐条提取——**新增科目零代码**。
- 同菜单承载：归档保留策略、上传大小上限、任务并发数、日志保留天数等系统参数（键值表 `sys_config`）。

### 2.8 文件存储与归档保留策略

```
data/
├── uploads/{module}/{yyyyMMdd}/{jobId}/      # 原始上传文件 / 直连查询快照CSV
├── results/{module}/{yyyyMMdd}/{jobId}/      # 结果文件
```

- 文件命名沿用业务规范（`基金资产与净值核对结果_YYYYMMDD.xlsx`）；同一业务日期重复执行自动追加版本号，不覆盖。
- **保留策略（平台自主实现）**：默认全量保留；系统配置中提供"自动清理"开关与"保留最近 N 天"参数（默认关闭）；清理任务每日低峰执行，**仅删除文件实体，保留数据库索引与统计记录**（历史趋势看板不受清理影响）；归档目录与平台库纳入每日备份。

### 2.9 任务调度设计

- APScheduler 持久化，重启不丢任务；cron 表达式避开整点/半点。**数据就绪时间可配置**：估值数据约每日 17:30 就绪，默认定时窗口 = 就绪时间 + 30 分钟缓冲（如 `0 18 * * 1-5`），就绪时间与缓冲分钟数均为系统参数。**初期交付默认手工触发核对，定时自动任务默认关闭，作为优化项按需启用**。
- 直连模式每日任务：按业务日期参数执行查询模板 → 核对 → 归档 → 站内通知；文件模式：监测目录路径可配置（当前无固定共享目录，申请到后改配置即生效），未就绪站内提醒人工上传。
- 失败自动重试 1 次，仍失败告警并保留完整日志。

## 三、客户端设计

### 3.1 菜单结构（V2.1）

```
├── 工作台（今日核对状态、异常预警、待办）
├── 数据核对中心
│   ├── 基金资产与净值核对   → 新建任务（文件/直连双模式）/ 结果页 / 历史页
│   ├── 估值价格核对         → 新建任务 / 结果页（按产品 Tab）/ 历史页
│   └── 银行间ID匹配         → 新建任务 / 结果页（三类标注统计）/ 历史页
├── 统计看板                 → 差异趋势 / 持续差异产品 / 任务健康度
├── 任务调度中心             → 定时任务 / 执行历史与日志
├── 数据源管理（管理员）      → 连接配置（测试连接）/ 查询模板
├── 规则配置中心（管理员）    → 映射规则 / 大宗产品清单 / 阈值 / 列模板
├── 报告归档中心             → 检索 / 下载 / 跨日对比
└── 系统管理（管理员）
    ├── 用户维护             → 用户列表 / 新增 / 重置密码 / 角色
    ├── 认证配置             → 本地/AD 切换与参数
    ├── 系统配置             → 科目取价规则 / 归档策略 / 系统参数
    └── 审计日志
```

### 3.2 关键页面交互（M1，对齐小程序操作体验）

1. **新建任务**：选择取数模式（文件上传 / 数据库查询）；文件模式拖拽上传两表（自动识别日期、支持多日期分组）；直连模式选择预置查询模板与业务日期。
2. **执行前确认**：展示生效配置摘要（映射 21 条、大宗 11 个、阈值 1.0%/0.5）；支持本次临时调阈值。
3. **执行**：进度条 + 实时日志区（WebSocket 推送，文案与小程序日志一致，如"正在映射去重...""精确匹配: N 条"）；校验失败时弹出与小程序一致的中文错误（列数不足/疑似选反）。
4. **结果页**：统计卡片（总记录/精确/模糊/未匹配/大宗产品数/差异>1%（非大宗））+ 明细表格（橙/红/黄行内标注，与 Excel 同色）+ 差异原因在线标注 + 下载 Excel（文件名 `基金资产与净值核对结果_YYYYMMDD.xlsx`）。

M2/M3 复用同一交互骨架，保证操作一致性。

## 五、自我论证与设计完善（资深运维 + 产品经理双视角）

> 按"先给结论、再自我驳斥、驳不倒的纳入方案"的方式，对关键决策做多角色论证。

**议题 1：已有 exe 小程序，为什么还要做平台？**
- 反方：小程序已满足每日核对，B/S 平台是过度工程。
- 驳斥：小程序解决的是"单人单机执行"，解决不了四个结构性问题——①规则漂移（config JSON 在个人电脑，多人多版本口径不一致）；②结果资产化（无归档检索、无跨日趋势，持续性差异产品靠人脑记）；③自动化（exe 无法定时+直连取数）；④内控审计（无用户体系、无操作留痕）。平台不是重写小程序，而是把它的引擎升级为共享服务。
- 结论：做平台成立；**保留 exe 作为应急兜底**（引擎同源、配置兼容，输出一致），已写入边界约束。

**议题 2：M1 引擎重写还是复用？**
- 反方：Web 服务化应顺便重构代码。
- 驳斥：验收 V1 要求"与小程序输出逐行一致"，任何重写都引入口径偏差风险且收益为零；小程序类设计本就是配置注入式，复用成本极低。
- 结论：原样复用 + 黄金样本回归测试，这是本项目口径不漂移的根本保障。

**议题 3：DB 直连会不会动到生产库？**
- 反方（安全/运维视角）：直连生产库风险高，不如坚持文件导出。
- 驳斥：四道防线（只读账号、SELECT 白名单、超时/行数限制、全量审计）可将风险压到与"人工用 PL/SQL Developer 查询"同等水平——而现状恰恰是人工天天在查。直连只是把已存在的人工只读访问规范化、留痕化。文件模式保留为兜底与并行验证手段。
- 遗留风险（如实标注）：①网络开通与只读账号申请依赖 DBA/厂商配合，是二期的外部依赖点；②SQL 模板质量决定取数正确性，**模板上线前必须用文件模式输出做并行核对验证**；③ thin 模式对个别老旧 Oracle 特性可能不兼容，预留 thick 切换。

**议题 4：认证默认本地还是 AD？**
- 反方：内控上 AD 更规范，应一步到位。
- 驳斥：AD 对接依赖域控参数与网络策略开通，周期不可控，会拖住一期上线；本地 bcrypt 认证 + 完整用户维护菜单已满足最小内控要求，且认证配置化后切 AD 零代码。
- 结论：默认本地、AD 可配置，符合"先跑起来、再规范化"的演进路径。

**议题 5：科目取价规则放"规则配置中心"还是"系统配置"？**
- 用户明确要求放系统配置菜单。该规则变更频率低、影响全局口径，与映射规则（业务口径、变更相对频繁）分菜单管理反而更合理：系统配置入口更深、仅管理员可见，降低误改概率。按用户要求落地，论证通过。

**议题 6：平台数据库选 MariaDB？**
- 反方：SQLite 更轻。
- 驳斥（V2.0 观点）：多人并发 + 定时任务 + 审计写入，SQLite 锁竞争明显；MariaDB 与团队 MySQL 运维经验一致。
- V2.0 结论：平台库 MariaDB，开发期可 SQLite 起步。**（此结论已被 V2.1 议题 7 正式反转，见下）**

**议题 7：无开发环境约束下平台库选 SQLite 还是 MariaDB？（对议题 6 结论的反转）**
- V2.1 新增硬约束：目标服务器无开发环境、不允许额外安装插件/软件。MariaDB 需独立安装部署，直接违背该约束；SQLite 内嵌于交付包，零安装、备份即复制文件。
- 对议题 6 并发论据的重新评估：平台用户为部门内个位数并发，写负载以任务结果与审计日志为主、读多写少，SQLite WAL 模式下完全足够；且数据访问层统一走 SQLAlchemy，后续负载增长时通过配置项切换 MariaDB，**切换零代码、双数据库表结构同源**。
- 结论（正式反转议题 6）：**默认内嵌 SQLite，保留 MariaDB 可配置切换能力**，特此记录。

**议题 8：后端打包 PyInstaller vs 绿色版 Python 运行时？**
- 反方：绿色版 Python + 依赖目录更透明、排错容易、还能现场改代码。
- 驳斥：绿色运行时目录庞大、入口脚本依赖 PATH 与文件关联，服务化注册与依赖一致性差；"现场改代码"恰是运维大忌（绕过版本管理与黄金样本回归）。PyInstaller 单 exe 锁定依赖版本、解压即用，符合"目标机零安装"硬约束；可调试性通过保留 console 启动模式 + 完整日志输出弥补。
- 结论：PyInstaller（备选 Nuitka，如需进一步压缩体积与防反编译，二期评估）。

**议题 9：前后端同进程托管是否违反"前后端隔离"？**
- 反方：后端 exe 直接托管前端静态文件，看起来是"合在一起"了。
- 驳斥：隔离要求是**工程与契约层面**的隔离——前后端分目录、独立构建、独立版本、仅 REST API + WebSocket 通信、无共享代码与状态；静态托管只是部署形态（等价于 Nginx 托管的简化），前端产物可随时改由 Nginx 托管而后端零改动。
- 结论：一期采用 exe 托管（减少组件、免安装 Nginx），架构上保留 Nginx 前置的可选部署形态。

**补充完善点（论证过程中新增进方案的）**：
1. 直连查询结果落一份**查询快照 CSV** 到归档（`recon_result_file` 类型=查询快照），保证直连任务也可重算复现、可与文件模式比对；
2. SQL 模板上线流程增加"文件模式并行验证"步骤（议题 3 遗留风险的闭环措施）；
3. exe 与平台共用配置口径：规则中心支持按小程序 config JSON 结构导入/导出；
4. 直连查询默认加业务日期参数与查询窗口约束，避免全表扫描压到生产库；
5. 交付包必须在**干净 Windows 虚拟机（无 Python/Node/任何运行时）** 冒烟通过方可发布（验收 V10）；
6. 打包机（开发机）是唯一需要开发环境的机器，打包流水线内嵌依赖审计与许可清单，避免目标机依赖任何外部组件。

## 八、目录结构与交付布局设计（V2.2 修订）

> 设计依据：经澄清，"独立"指**存放路径的独立隔离**——文档与代码不混放、分文件夹管理、整洁有序。本章在此基础上从软件工程角度给出合理布局，原则是"路径分清楚、收益拿得到、不引入过度工程"。

### 8.1 开发侧目录布局（文档与代码分文件夹）

```
O32-OpsPlatform/                  # 项目根目录（内网版本管理）
├── docs/                         # ★ 文档专区（与代码完全分开放置）
│   ├── 01-需求/                  # 需求文档（按版本存档，如 需求文档_V2.2.md）
│   ├── 02-方案/                  # 落地方案、接口契约（OpenAPI 导出）
│   ├── 03-手册/                  # 部署手册、操作手册
│   └── 04-测试/                  # 黄金样本说明、回归测试报告
├── client/                       # ★ 前端代码（独立文件夹、独立构建）
├── server/                       # ★ 后端代码（独立文件夹、独立构建）
│   ├── app/  tests/  packaging/
├── samples/                      # 黄金样本（脱敏构造数据，不含真实生产数据）
└── README.md                     # 仅一页：项目简介 + 构建/部署指引
```

- 文档只进 `docs/`，代码只进 `client/`、`server/`，各就其位；编号前缀保证文件夹里排序整洁。
- 文档按版本存档，与交付包版本号对应（如交付包 v1.0.0 对应的需求/方案文档存档副本）。
- `README.md` 保持一页，详细内容一律放 `docs/`。

### 8.2 服务器侧部署布局（程序、数据、归档分文件夹）

```
D:\O32OpsPlatform\
├── app\                          # 程序（后端 exe + 前端静态文件）——升级只替换这里
│   ├── server\o32-server.exe
│   └── web\                      # 前端预编译静态文件
├── data\                         # 平台数据（SQLite 库、平台配置）——升级永不覆盖
├── archive\                      # 归档（原始上传、查询快照、核对报告）
├── logs\                         # 运行日志
├── backup\                       # 每日备份（data+archive 压缩包，保留最近 30 份）
├── app_旧版本\                   # 保留最近 2 个旧版程序目录，用于回滚
├── nssm\  install.bat  uninstall.bat
└── 版本.txt                      # 当前版本、发布日期、变更摘要
```

- 升级 = 替换 `app\`；回滚 = 服务指向旧程序目录重启；数据、归档、日志、备份均不受影响。
- 此布局同时满足"配置热更新不重新打包"（业务规则存放在 `data\` 平台库中）与"整包免安装交付"。

### 8.3 前后端协作边界（路径分离自然带来的工程收益，无额外管理成本）

- 前后端分文件夹后，天然形成"只通过接口通信"的边界：后端 FastAPI 自动导出 OpenAPI 接口文档，存档至 `docs/02-方案/`，前端照契约开发。
- 前端不直连任何数据库；后端不读取前端源码目录（仅托管其构建产物）。这些是布局的自然结果，不需要额外的流程管控。

### 8.4 无开发环境交付设计

目标服务器与操作终端**没有任何开发环境，且不允许额外安装插件/运行时**。所有依赖必须在打包机上预编译/预置进交付包：

| 组件 | 交付形态 | 预置内容 |
|---|---|---|
| 后端 | PyInstaller 独立 exe | Python 3.11 运行时 + 全部第三方库；数据库驱动全选免安装版（oracledb thin 免 Oracle Client、pymysql 纯 Python、pymssql 自带 FreeTDS、psycopg2-binary 自带 libpq） |
| 前端 | `vite build` 预编译静态文件 | 随后端包交付，由 exe 内 StaticFiles 托管；浏览器零插件（不使用任何需浏览器插件的技术） |
| 平台库 | SQLite 内嵌 | 首次启动自动建库建表 + 初始数据（21 条映射、11 个大宗产品、默认阈值、科目取价规则 2 条、初始管理员）；备份=复制文件 |
| 服务化 | NSSM 单 exe 随包 | `install.bat` 注册 Windows 服务、开机自启；`uninstall.bat` 卸载 |
| 部署包 | `o32-ops-platform-vX.Y.Z.zip` | 解压即用：目录约定 `app/`（exe+静态文件）、`data/`（SQLite+归档，独立存放便于升级保留）、`logs/`、`nssm/`、安装脚本、部署手册 |

- **数据与程序分离**：`data/`（SQLite 库文件、归档文件、平台配置）独立于 `app/` 程序目录，升级=整包替换 `app/`，数据与配置原样保留。
- **配置热更新**：业务规则修改不触发重新打包（沿用《做事标准》：配置与代码分离）；代码逻辑变更才重新打包发版。
- **冒烟验证**：每期交付包在干净 Windows 虚拟机（无 Python/Node/任何运行时）完成全流程冒烟（安装→启动→登录→M1 核对→下载报告）方可发布（验收 V10）。
- **回滚机制**：保留最近 2 个版本 `app/` 目录，回滚=切换服务指向旧目录并重启服务。
